Bottom line: The most dangerous defects are in the mitigation pipeline. The process-kill mechanism is entirely non-functional (C-01), restoration races a live ransomware process (C-02), and Mode A's kernel socket is a local privilege escalation vector (C-04). SHIELD's core guarantee — detect, kill, restore — only works for screen-locker ransomware caught by LockerGuard. For all other families the detection may fire correctly but the response fails.

The cryptographic foundation (AES-256-GCM, Keystore, hash chain, TEE integrity gate) is production-quality and should be preserved. Mode A's architecture is the correct long-term design; its three blocking issues are fixable without redesign.

Critical Issues

C-01 CRITICAL
Process Kill Does Nothing — Ransomware Survives and Re-Encrypts Restored Files
UnifiedDetectionEngine.java · killMaliciousProcess()
Problem

killMaliciousProcess() calls ActivityManager.killBackgroundProcesses() inside a comment-only for-loop. killBackgroundProcesses() is explicitly documented to do nothing to foreground services. Every real-world Android ransomware runs as a foreground service to survive exactly this call. As a result the process is still running when finalizeMitigationAndRestore() fires 1 second later, and all just-restored files are immediately re-encrypted. The README's "automated data recovery" guarantee is therefore false for any foreground-service ransomware.

Root Cause

The loop body contains only a Log.w() call; the actual kill is attempted with the wrong API. killBackgroundProcesses() only terminates background processes that the OS considers safe to kill — foreground services are explicitly protected.

Solution

Replace with a four-layer termination sequence: (1) Use AccessibilityService.performGlobalAction(GLOBAL_ACTION_HOME + GLOBAL_ACTION_BACK) to navigate away from the locker UI immediately. (2) Force-stop via the App Info deep-link Intent (ACTION_APPLICATION_DETAILS_SETTINGS) prompting the user for one tap. (3) Broadcast ACTION_KILL_PID to ModeAService/JNI which calls nativeSendKill(pid) — this uses bpf_send_signal(SIGKILL) from kernel context and cannot be bypassed by a foreground service. (4) Do not call finalizeMitigationAndRestore() until the kill is confirmed via process-death polling or the user has manually force-stopped the app.

Code Fix
// Step 1 – navigate away (LockerShieldService already does this)

// Step 2 – open App Info so user can tap Force Stop
Intent appInfo = new Intent(Settings.ACTION_APPLICATION_DETAILS_SETTINGS);
appInfo.setData(Uri.parse("package:" + packageName));
appInfo.addFlags(Intent.FLAG_ACTIVITY_NEW_TASK);
context.startActivity(appInfo);

// Step 3 – kernel kill via Mode A JNI (if Mode A is running)
Intent killIntent = new Intent(ModeAService.ACTION_KILL_PID);
killIntent.putExtra("pid", resolvedPid);
context.sendBroadcast(killIntent);

// Step 4 – only restore AFTER confirming process is gone
executor.submit(() -> {
    if (waitForProcessDeath(packageName, 10_000)) {
        snapshotManager.stopAttackTracking();
        snapshotManager.performAutomatedRestore();
    } else {
        // Notify user: manual Force Stop required before restore
        showManualStopRequiredAlert(packageName);
    }
});
C-02 CRITICAL
Restore Race Condition — 1-Second Delay Is Insufficient
UnifiedDetectionEngine.java · finalizeMitigationAndRestore()
Problem

finalizeMitigationAndRestore() uses detectionHandler.postDelayed(..., 1000) — a hard-coded 1-second wait — before calling performAutomatedRestore(). Because the process kill fails (C-01), this is simply a 1-second pause before the ransomware overwrites all restored files. Even if the kill succeeded, 1 second is not guaranteed to be sufficient for the OS to reclaim file handles.

Root Cause

The restore sequence is time-based rather than state-based. A fixed delay cannot reliably reflect whether the malicious process has actually terminated.

Solution

Replace the time-based delay with a process-death poll. After issuing the kill sequence, poll /proc/<pid>/status every 200 ms for up to 10 seconds. Only invoke performAutomatedRestore() once the PID directory disappears (process confirmed dead), or fall back to prompting the user for manual Force Stop before restoring.

Code Fix
private boolean waitForProcessDeath(String pkg, long timeoutMs) {
    long deadline = System.currentTimeMillis() + timeoutMs;
    while (System.currentTimeMillis() < deadline) {
        if (!isProcessRunning(pkg)) return true;
        try { Thread.sleep(200); } catch (InterruptedException e) { break; }
    }
    return false;  // timed out — process still alive
}

private boolean isProcessRunning(String pkg) {
    ActivityManager am = (ActivityManager)
        context.getSystemService(Context.ACTIVITY_SERVICE);
    if (am == null) return false;
    for (ActivityManager.RunningAppProcessInfo p : am.getRunningAppProcesses()) {
        if (Arrays.asList(p.pkgList).contains(pkg)) return true;
    }
    return false;
}

// Replace postDelayed(1000) with:
executor.submit(() -> {
    boolean dead = waitForProcessDeath(suspectPackage, 10_000);
    if (dead) snapshotManager.performAutomatedRestore();
    else showManualStopRequiredAlert(suspectPackage);
});
C-03 CRITICAL
EXPECTED_SIGNATURE_HASH = null — APK Signature Check Always Skips (Debug) or Breaks Service (Release)
SecurityUtils.java · EXPECTED_SIGNATURE_HASH constant
Problem

The constant is set to null. In debug builds, verifySignature() logs the actual hash and returns true (check skipped). In release builds, the improved code returns false (fails closed), which means checkSecurity() reports isSafe = false and the integrity gate in onStartCommand() halts the service — breaking the genuine APK. In both cases the tamper-detection guarantee is absent.

Root Cause

The developer set a null placeholder with the comment "Set in production" but never set it. The build pipeline has no enforcement step to catch this before release.

Solution

After signing the release APK, run keytool -printcert -jarfile release.apk | grep SHA256 and paste the lowercase hex string into EXPECTED_SIGNATURE_HASH. Add a Gradle task that asserts the constant is non-null before any assembleRelease build.

Code Fix
// In SecurityUtils.java — set your actual cert hash:
private static final String EXPECTED_SIGNATURE_HASH =
    "a3f8c2d1e4b7...";  // paste SHA-256 output from keytool here

// In build.gradle — block release builds if hash is still null:
task checkSignatureHash {
    doLast {
        def src = file('src/main/java/.../SecurityUtils.java').text
        if (src.contains('EXPECTED_SIGNATURE_HASH = null')) {
            throw new GradleException(
                'EXPECTED_SIGNATURE_HASH must be set before release build')
        }
    }
}
assembleRelease.dependsOn checkSignatureHash
C-04 CRITICAL
Mode A KILL Socket World-Writable — Any App Can Kill Arbitrary PIDs as Root
modea/ daemon (native) · socket setup
Problem

The root daemon creates the UNIX domain socket at /data/local/tmp/shield_modea.sock with chmod 0666 and performs no SO_PEERCRED peer-credential validation before honouring KILL commands. Any installed app on the device can connect to this socket and send a kill command for any PID, including system_server — effectively giving every app unrestricted root-level process termination. This is a local privilege escalation (LPE) vulnerability.

Root Cause

Socket was created in a development/test configuration where open permissions were convenient. Peer authentication was never added when the socket protocol was designed.

Solution

Change socket permissions to 0600 (owner-only). In the daemon's accept() loop, call getsockopt(fd, SOL_SOCKET, SO_PEERCRED, &cred, &len) and verify cred.uid matches SHIELD's expected UID. Reject all other connections. Move the socket from the world-accessible /data/local/tmp/ to SHIELD's private /data/data/com.dearmoon.shield/ directory.

Code Fix
// In daemon C++ — after bind():
chmod(SOCKET_PATH, 0600);  // owner-only, NOT 0666

// In accept() loop — verify peer UID before honouring any command:
struct ucred cred;
socklen_t len = sizeof(cred);
if (getsockopt(client_fd, SOL_SOCKET, SO_PEERCRED, &cred, &len) < 0 ||
    cred.uid != SHIELD_EXPECTED_UID) {
    close(client_fd);
    continue;  // reject unauthorised peer silently
}

// Move socket path to app-private directory:
#define SOCKET_PATH "/data/data/com.dearmoon.shield/shield_modea.sock"
C-05 CRITICAL
Mode A BURST_WRITE_THRESHOLD=1 and uid_is_app() Removed — Permanent Kernel Flood Mode
modea/shield_bpf.c · eBPF filter logic
Problem

The eBPF probe has BURST_WRITE_THRESHOLD set to 1 (fires on every single file write) and the uid_is_app() filter that restricts events to UIDs 10000–19999 (app range) has been commented out. Every kernel file-write by every process — including system_server, mediaserver, zygote — is forwarded to the Java detection engine at full kernel speed. The result is tens of thousands of events per second: the app becomes unresponsive, the battery drains in minutes, and detection results are meaningless noise.

Root Cause

These filters were disabled during debugging and never restored before commit. A development configuration was accidentally treated as production.

Solution

Restore uid_is_app(): only forward events where uid >= 10000 && uid <= 19999. Raise BURST_WRITE_THRESHOLD back to 20 (fires only when 20+ writes occur within a 500 ms window). Add a per-UID rate-limiter BPF hash map to enforce the threshold at kernel level before any userspace overhead.

Code Fix
// In shield_bpf.c — restore both filters:
#define BURST_WRITE_THRESHOLD 20    // restore from 1
#define UID_APP_MIN           10000
#define UID_APP_MAX           19999

// In write_probe handler:
u32 uid = bpf_get_current_uid_gid() & 0xFFFFFFFF;
if (uid < UID_APP_MIN || uid > UID_APP_MAX) return 0;  // skip non-app UIDs

// Per-UID burst rate-limiter via BPF hash map:
u64 *count = bpf_map_lookup_elem(&write_count_map, &uid);
if (!count) {
    u64 zero = 0;
    bpf_map_update_elem(&write_count_map, &uid, &zero, BPF_NOEXIST);
    return 0;
}
if (*count < BURST_WRITE_THRESHOLD) {
    __sync_fetch_and_add(count, 1);
    return 0;  // below threshold — do not forward
}
// threshold exceeded — forward event to userspace
C-06 CRITICAL
Mode A Uses system() with UID 0 — Command Injection as Root
modea/ daemon (native) · BPF filesystem mount
Problem

The daemon calls system("mount -t bpf none /sys/fs/bpf") to mount the BPF filesystem. The system() C function invokes /bin/sh -c with the provided string. Running system() as UID 0 with any environment-influenced argument is a command injection vulnerability. An attacker who controls environment variables (PATH, IFS) or /bin/sh can escalate to arbitrary root code execution.

Root Cause

Convenience: system() is easier to call than execve(). The developer did not consider that UID 0 + system() is always a command injection risk regardless of the argument string.

Solution

Replace system() with execve() using a NULL-terminated argv array — no shell involved. First check if /sys/fs/bpf is already mounted (it usually is on Android 10+), making the call unnecessary in most cases.

Code Fix
// WRONG — shell injection risk at UID 0:
system("mount -t bpf none /sys/fs/bpf");

// CORRECT — check if already mounted first:
struct statfs sfs;
if (statfs("/sys/fs/bpf", &sfs) == 0 && sfs.f_type == BPF_FS_MAGIC) {
    // Already mounted — skip
} else {
    // Use execve with explicit args — no shell:
    char *argv[] = {"/system/bin/mount", "-t", "bpf",
                    "none", "/sys/fs/bpf", NULL};
    pid_t pid = fork();
    if (pid == 0) {
        execve(argv[0], argv, environ);
        _exit(1);  // execve failed
    }
    int status;
    waitpid(pid, &status, 0);
}

High Severity Issues

H-01 HIGH
intentionally_stopped Flag Race — Service Restarts Even After User Stops It
ShieldProtectionService.java · onDestroy() / ShieldWatchdogService.java
Problem

onDestroy() sets intentionally_stopped = true in SharedPreferences via apply() (asynchronous), then immediately broadcasts RESTART_SERVICE. SharedPreferences apply() writes are not guaranteed to be flushed before the watchdog's next checkMainService() fires within the 5-second window. The result: the watchdog reads a stale false value and restarts the service the user just stopped.

Root Cause

Two restart mechanisms (broadcast + watchdog) without a consistent synchronisation point, combined with the asynchronous nature of apply().

Solution

Use .commit() (synchronous) instead of .apply() when setting intentionally_stopped. Remove the RESTART_SERVICE broadcast from onDestroy() entirely — the watchdog already handles restarts. When the watchdog reads the flag as true, it should call stopSelf() to stop itself too.

Code Fix
// In ShieldProtectionService.onDestroy() — synchronous write:
getSharedPreferences("ShieldPrefs", Context.MODE_PRIVATE)
    .edit().putBoolean("intentionally_stopped", true).commit(); // commit() not apply()

// Remove the restart broadcast entirely — watchdog handles it:
// sendBroadcast(new Intent("com.dearmoon.shield.RESTART_SERVICE")); // DELETE THIS

// In ShieldWatchdogService.checkMainService():
if (intentionallyStopped) {
    Log.d(TAG, "Intentionally stopped — shutting down watchdog too");
    stopSelf();  // watchdog stops itself
    return;
}
H-02 HIGH
MediaStoreCollector Disabled — Blind to ContentResolver-Based Ransomware
ShieldProtectionService.java · initializeCollectors()
Problem

MediaStoreCollector is fully implemented but never instantiated or started in initializeCollectors(). Ransomware families that use ContentResolver to write encrypted files (common for media targeting .jpg, .mp4, .docx etc.) bypass FileObserver entirely and are completely invisible to SHIELD. The MediaStore onChange() callback is the only way to catch ContentResolver-based writes without root.

Root Cause

Collector was disabled at some point, likely due to ContentResolver spam, and never re-enabled. The existing 2-second debounce in the implementation already solves the spam problem.

Solution

Instantiate MediaStoreCollector in initializeCollectors() and call startWatching(). Wire it to both detectionEngine and snapshotManager identically to how RecursiveFileSystemCollector is wired. Store the reference and call stopWatching() in onDestroy().

Code Fix
// In initializeCollectors() — add after recursive collectors:
mediaStoreCollector = new MediaStoreCollector(this, storage, detectionEngine);
mediaStoreCollector.startWatching();
Log.i(TAG, "MediaStore monitoring enabled");

// In onDestroy():
if (mediaStoreCollector != null) {
    mediaStoreCollector.stopWatching();
}

// Add field to ShieldProtectionService:
private MediaStoreCollector mediaStoreCollector;
H-03 HIGH
SPRT H1 Rate (5.0 files/sec) Never Validated — False Positive Rate Unknown
SPRTDetector.java · RANSOMWARE_RATE / NORMAL_RATE constants
Problem

RANSOMWARE_RATE = 5.0 and NORMAL_RATE = 0.1 were set heuristically and never validated against real Android device baseline activity. Normal Android background activity (Sync adapters, Play Store, camera scans) can generate 2–10 file writes per second during normal use. If the benign baseline H0 = 0.1 files/sec is too low (which it almost certainly is), SPRT fires ACCEPT_H1 during legitimate background activity, contributing 30 points to every detection score and causing false alerts every time the device syncs.

Root Cause

No empirical baseline measurement was performed. Values were chosen as round numbers that "seemed reasonable" without measuring real device telemetry.

Solution

Instrument a test build to log inter-event deltas and events-per-second across 10+ real devices over 24 hours of normal use. Fit a Poisson distribution to derive empirical H0. Use the 99th-percentile benign burst rate as H0. Until measured, raise MIN_SAMPLES_FOR_H1 from 3 to 10 as a conservative guard and add telemetry logging to measure the baseline.

Code Fix
// Temporary conservative guard until baseline is empirically measured:
private static final int MIN_SAMPLES_FOR_H1 = 10;  // up from 3

// Target empirical values after measurement (illustrative — measure first):
// private static final double NORMAL_RATE     = 2.0;  // measured P99 benign
// private static final double RANSOMWARE_RATE = 25.0; // observed ransomware burst

// Add telemetry to measure baseline in instrumented builds:
if (BuildConfig.DEBUG) {
    Log.d(TAG, "SPRT_METRIC delta_ms=" + deltaMs
        + " events_per_sec=" + String.format("%.2f", 1000.0 / deltaMs)
        + " llr=" + String.format("%.4f", logLikelihoodRatio)
        + " state=" + currentState);
}
H-04 HIGH
VPN Started Without Checking Permission — Silent Failure
ShieldProtectionService.java · startNetworkMonitoring()
Problem

startNetworkMonitoring() calls startForegroundService(vpnIntent) unconditionally. VpnService.prepare(context) must be called first and, if it returns a non-null Intent, the user must complete a system permission dialog. If permission has not been granted, establish() in NetworkGuardService returns null and the service silently calls stopSelf(). SHIELD runs with no network layer and the user has no idea.

Root Cause

VPN permission is unusual — it requires an Activity-level dialog and cannot be granted silently. This requirement was overlooked when the service start flow was designed.

Solution

Before calling startNetworkMonitoring(), call VpnService.prepare(context) from an Activity context. If the result Intent is non-null, launch it with startActivityForResult(). Start the VPN service only after the user grants permission in onActivityResult(). Show a persistent UI indicator when the VPN layer is not active.

Code Fix
// In MainActivity.java — check VPN permission before starting SHIELD:
private static final int REQUEST_VPN_PERMISSION = 1001;

void startShieldWithVpnCheck() {
    Intent vpnPermIntent = VpnService.prepare(this);
    if (vpnPermIntent != null) {
        // System dialog required — user must grant
        startActivityForResult(vpnPermIntent, REQUEST_VPN_PERMISSION);
    } else {
        // Already granted
        startShieldProtectionService();
    }
}

@Override
protected void onActivityResult(int requestCode, int resultCode, Intent data) {
    if (requestCode == REQUEST_VPN_PERMISSION) {
        if (resultCode == RESULT_OK) {
            startShieldProtectionService();
        } else {
            showVpnPermissionDeniedWarning();  // inform user — network layer inactive
        }
    }
}
H-05 HIGH
VPN Blocklist Uses 2016-Era RAT Ports — Ineffective Against Modern HTTPS/443 C2
NetworkGuardService.java · shouldBlockConnection() · isTorExitNode()
Problem

The blocklist checks only ports 4444/5555/6666/7777 (Metasploit/RAT defaults from 2015–2016) and a hard-coded list of 10 IP prefixes. Modern Android ransomware (Sova, SharkBot, Xenomorph variants) exclusively uses HTTPS on port 443 with Telegram bot API, Pastebin, or GitHub Gist as C2 channels. The blocklist misses 100% of these. The only effective defence is the all-traffic emergency block — but that only fires after detection, not before.

Root Cause

The blocklist was written using knowledge of old Android RAT toolkits. Modern mobile malware deliberately uses legitimate cloud infrastructure to evade static IP/port blocklists.

Solution

Replace the static port/IP blocklist as the primary defence. Instead: (1) Set Quad9 (9.9.9.9) or CleanBrowsing security DNS as the VPN's upstream resolver — blocks known malicious domains at resolution time. (2) Document clearly that the all-traffic emergency block post-detection is the only fully reliable isolation mechanism. (3) Optionally add TLS SNI inspection for known C2 domain patterns.

Code Fix
// In establishVpnInterface() — replace Google DNS with threat-blocking DNS:
builder.addDnsServer("9.9.9.9");          // Quad9 — blocks malicious domains
builder.addDnsServer("149.112.112.112");  // Quad9 secondary

// Or CleanBrowsing security filter:
// builder.addDnsServer("185.228.168.9");
// builder.addDnsServer("185.228.169.9");

// NOTE: Port 4444/5555 blocklist only catches legacy RAT traffic.

M-01 MEDIUM
Entropy Allowlist Includes .docx/.xlsx/.pdf — High-Value Ransomware Targets Invisible to Entropy Scoring
EntropyAnalyzer.java · NATURALLY_HIGH_ENTROPY_EXTENSIONS
Problem

The allowlist skips entropy and KL-divergence analysis for .docx, .xlsx, .pptx, .pdf, .jpg, .mp4, and many more. While these formats do have naturally high entropy, they are also the primary targets of document-encryption ransomware (Dharma, LockBit, Cl0p). When ransomware encrypts these files, calculateEntropy() returns 0.0, contributing 0 points to the confidence score. Detection relies entirely on SPRT file rate alone for these types — which is often insufficient.

Root Cause

The allowlist was designed to eliminate false positives from compressed/media formats but was applied too broadly, covering high-value document formats that ransomware explicitly targets.

Solution

Remove .docx, .xlsx, .pptx, .odt, .ods, .odp, and .pdf from the allowlist. Implement a magic-byte pre-check: DOCX/XLSX/PPTX start with PK\x03\x04 (ZIP); if this header is missing, the file has been replaced with encrypted data and should be scored normally. PDFs start with %PDF — if missing, score normally.

Code Fix
// Remove from NATURALLY_HIGH_ENTROPY_EXTENSIONS:
// ".docx", ".xlsx", ".pptx", ".odt", ".ods", ".odp", ".pdf"

// Add magic-byte structural validation:
private boolean hasExpectedMagicBytes(File file) {
    try (FileInputStream fis = new FileInputStream(file)) {
        byte[] magic = new byte[4];
        if (fis.read(magic) < 4) return false;
        // DOCX/XLSX/PPTX are ZIP: PK\x03\x04
        if (magic[0]==0x50 && magic[1]==0x4B &&
            magic[2]==0x03 && magic[3]==0x04) return true;
        // PDF: %PDF
        if (magic[0]==0x25 && magic[1]==0x50 &&
            magic[2]==0x44 && magic[3]==0x46) return true;
        return false;  // structure destroyed — score entropy normally
    } catch (Exception e) { return false; }
}

// In calculateEntropy(): if !hasExpectedMagicBytes(file) → proceed with scoring
// If magic bytes intact → return 0.0 (natural entropy, skip scoring)
M-02 MEDIUM
SPRT ACCEPT_H0 Reset Leaks Phantom Behavior Scores Into Next Window
UnifiedDetectionEngine.java · analyzeFileEvent() / SPRTDetector.java · reset()
Problem

When SPRT reaches ACCEPT_H0 and reset() is called, the LLR counter is cleared. However, file events logged to the EventDatabase during this window remain and are queried by BehaviorCorrelationEngine.queryRecentFileEvents() with a 5-second look-back. This means an ACCEPT_H0 followed by a new burst of events produces a high behaviorScore (from lingering database entries), potentially pushing the total score across the 40-point attack-tracking threshold with no real ransomware signal.

Root Cause

The SPRT state machine and the behavior correlation window are not synchronised. SPRT resets its internal counter but the database is not informed of the reset boundary.

Solution

Track a SPRT epoch counter that increments on every reset(). Store the reset timestamp. In BehaviorCorrelationEngine, filter file events to only those that occurred after the last SPRT reset, so old pre-reset events do not pollute the next window's behavior score.

Code Fix
// In SPRTDetector — track reset timestamp:
private long lastResetTimestamp = System.currentTimeMillis();
private int epoch = 0;

public synchronized void reset() {
    logLikelihoodRatio = 0.0;
    sampleCount = 0;
    currentState = SPRTState.CONTINUE;
    lastResetTimestamp = System.currentTimeMillis();
    epoch++;
}

public long getLastResetTimestamp() { return lastResetTimestamp; }

// In BehaviorCorrelationEngine — respect reset boundary:
long queryStart = Math.max(windowStart, sprtDetector.getLastResetTimestamp());
List<JSONObject> fileEvents =
    database.queryEventsSince("FILE_SYSTEM", queryStart, 100);
M-03 MEDIUM
Large File Hash Returns "LARGE_FILE" String — Hash Chain Integrity Broken for Files > 50 MB
SnapshotManager.java · calculateHash() / RestoreEngine.java · calculateQuickHash()
Problem

For files larger than 50 MB, calculateHash() returns the literal string "LARGE_FILE" instead of a real hash. This string is stored as sha256Hash in the hash chain. Any ransomware targeting large files (video files, databases) has its pre-attack hash recorded as "LARGE_FILE", making the chain comparison trivially defeatable. Two entirely different files have the same "hash", corrupting chain integrity for the entire chain.

Root Cause

Hashing large files was considered too slow and a placeholder was left in. The impact on chain integrity was not considered.

Solution

Remove the 50 MB ceiling entirely — use streaming MessageDigest for all files up to 500 MB. For files over 500 MB, use a partial hash: hash the first 1 MB + last 1 MB + file size encoded as 8 bytes. This is sufficient to detect ransomware replacement (which always changes the entire file content) while keeping computation bounded.

Code Fix
private String calculateHash(File file) throws Exception {
    MessageDigest digest = MessageDigest.getInstance("SHA-256");
    long size = file.length();
    try (FileInputStream fis = new FileInputStream(file)) {
        if (size <= 500L * 1024 * 1024) {
            // Full streaming hash for files up to 500 MB
            byte[] buffer = new byte[65536];
            int n;
            while ((n = fis.read(buffer)) != -1) digest.update(buffer, 0, n);
        } else {
            // Partial hash for very large files: first 1MB + last 1MB + size
            byte[] buf = new byte[1024 * 1024];
            int n = fis.read(buf);
            if (n > 0) digest.update(buf, 0, n);
            long skipTo = size - buf.length;
            fis.skip(skipTo - n);
            n = fis.read(buf);
            if (n > 0) digest.update(buf, 0, n);
            // Include file size to distinguish files of different lengths
            digest.update(ByteBuffer.allocate(8).putLong(size).array());
        }
    }
    byte[] hash = digest.digest();
    StringBuilder sb = new StringBuilder();
    for (byte b : hash) sb.append(String.format("%02x", b));
    return sb.toString();
}

L-01 LOW
ShieldWatchdogService Uses Deprecated getRunningServices() — Unreliable on API 26+
ShieldWatchdogService.java · isServiceRunning()
Problem

ActivityManager.getRunningServices() was deprecated in API 26 and explicitly documented to return only the calling app's own services on API 26+. On modern devices isServiceRunning(ShieldProtectionService.class) always returns true because ShieldProtectionService IS in the calling package, so the watchdog never detects a genuine service death and never triggers a restart.

Solution

Replace the service-running check with a SharedPreferences heartbeat: ShieldProtectionService writes a timestamp every 30 seconds; ShieldWatchdogService considers the service dead if the heartbeat is older than 60 seconds. Reliable across all API levels.

Code Fix
// In ShieldProtectionService — write heartbeat every 30s:
private void scheduleHeartbeat() {
    heartbeatHandler.postDelayed(() -> {
        getSharedPreferences("ShieldPrefs", MODE_PRIVATE).edit()
            .putLong("last_heartbeat", System.currentTimeMillis()).apply();
        scheduleHeartbeat();
    }, 30_000);
}

// In ShieldWatchdogService — check heartbeat age instead of getRunningServices():
private boolean isServiceAlive() {
    long last = getSharedPreferences("ShieldPrefs", MODE_PRIVATE)
        .getLong("last_heartbeat", 0);
    return System.currentTimeMillis() - last < 60_000;
}

L-02 LOW
HoneyfileCollector Grace-Period Uses Global Timestamp — Misses Per-File Restart Events
HoneyfileCollector.java · HoneyfileObserver.onEvent()
Problem

The grace period is measured from a single class-level lastHoneyfileCreationTime set once for the whole batch. On service restart with pre-existing honeyfiles, lastHoneyfileCreationTime is 0, making the grace period 0 ms and causing SHIELD to log its own startup file-access as a ransomware event. If honeyfile creation spans > 5 seconds (slow storage), files created last may not benefit from the grace period at all.

Solution

Track lastModified per-file within each HoneyfileObserver. If the event timestamp is within 5 seconds of the file's own lastModified time, treat it as self-creation and skip.

Code Fix
// Per-file grace period — store each file's own creation time:
private class HoneyfileObserver extends FileObserver {
    private final String filePath;
    private final long fileCreationTime;  // per-file, not class-level

    HoneyfileObserver(String path) {
        super(path, OPEN | MODIFY | DELETE | CLOSE_WRITE);
        this.filePath = path;
        this.fileCreationTime = new File(path).lastModified();  // reads actual file mtime
    }

    @Override
    public void onEvent(int event, String path) {
        long age = System.currentTimeMillis() - fileCreationTime;
        if (age < CREATION_GRACE_PERIOD_MS) {
            Log.d(TAG, "Grace period active for: " + filePath); return;
        }
        // ... rest of event handling
    }
}
L-03 LOW
Honeyfile Names Are Predictable — Sophisticated Ransomware Can Skip Them
HoneyfileCollector.java · createHoneyfiles() · honeyfileNames array
Problem

The six honeyfile names (IMPORTANT_BACKUP.txt, PRIVATE_KEYS.dat, etc.) are static constants compiled into the APK. A ransomware author can decompile SHIELD, extract the list, and add them to a skip list — silently disabling SHIELD's most reliable early-warning signal. Once public, threat intelligence would distribute this skip list to all new ransomware variants within days.

Solution

Generate honeyfile names at install time using a hash of the device ID + install timestamp. Store the generated names in Keystore-protected SharedPreferences. This makes names unique per device and unpredictable without physical device access.

Code Fix
private String generateHoneyfileName(String dir, int index) throws Exception {
    PackageInfo pi = context.getPackageManager()
        .getPackageInfo(context.getPackageName(), 0);
    String seed = Build.FINGERPRINT + dir + index + pi.firstInstallTime;
    byte[] hash = MessageDigest.getInstance("SHA-256").digest(
        seed.getBytes(StandardCharsets.UTF_8));
    // Take first 8 hex chars as filename prefix — unique per device
    StringBuilder sb = new StringBuilder();
    for (int i = 0; i < 4; i++) sb.append(String.format("%02X", hash[i]));
    String[] exts = {".txt", ".dat", ".bin", ".bak", ".key", ".db"};
    return sb.toString() + exts[index % exts.length];
}

// Store generated names at first run, reload on subsequent runs:
SharedPreferences prefs = getEncryptedPrefs();  // Keystore-backed
String key = "honeyfile_" + dirHash + "_" + index;
String name = prefs.getString(key, null);
if (name == null) {
    name = generateHoneyfileName(dir, index);
    prefs.edit().putString(key, name).apply();
}
