App Identity

What This App Is
& True Intended Behaviour

SHIELD is a rootless Android ransomware detection-and-mitigation system. Its mission: monitor, detect, isolate, and recover from ransomware attacks entirely on-device — no cloud, no server, no root required for the primary mode.

Core Architecture

Two modes: Mode B (primary) runs on every device using FileObserver, VPN packet capture, and Accessibility Service. Mode A (research extension) uses eBPF kernel probes on rooted devices for accurate PID/UID attribution. Mode A lives in modea/ and is not merged into the main APK.

01
📡
COLLECT
Fully Wired

Four parallel data sources: FileSystemCollector (FileObserver, depth-8 recursive), HoneyfileCollector (decoys in every dir), NetworkGuardService (VPN, IPv4+IPv6), LockerShieldService (Accessibility). All four write to SQLite on every event.

Working, Uncalibrated

Shannon entropy (0–40 pts) + KL divergence (0–30 pts) + SPRT Poisson test (0–30 pts) + Behavior correlation (0–30 pts). Max 130. Threshold: ≥70 = HIGH RISK. Extension allowlist + magic-byte check prevents false positives on document files.

03

Network ✅ Kill ⚠️

Emergency VPN mode drops all outgoing packets — genuinely effective. killBackgroundProcesses() has zero effect on foreground services. App Info is opened for user Force Stop. Mode A can issue root SIGKILL when deployed.

Significant Fixes — Detail

Fixed — F-01 Critical
waitForProcessDeath() now receives the actual package name

The signature of finalizeMitigationAndRestore() now takes String suspectPackageName and threads it through to waitForProcessDeath(suspectPackageName, 10_000). isProcessRunning(pkg) now correctly polls Arrays.asList(p.pkgList).contains(pkg) with a real string. If the process dies, restore proceeds. If the 10-second timeout elapses with the process still alive, showManualStopRequiredAlert() is called instead — no race condition. Additionally, sprtDetector.reset() is called on successful process death, clearing the ACCEPT_H1 latch (fix for F-03 bundled here).

// Before (broken): boolean dead = waitForProcessDeath(null, 10_000); // After (fixed): private void finalizeMitigationAndRestore(String suspectPackageName) { ... boolean dead = waitForProcessDeath(suspectPackageName, 10_000); if (dead) { snapshotManager.stopAttackTracking(); sprtDetector.reset(); // clears ACCEPT_H1 latch snapshotManager.performAutomatedRestore(); } else { showManualStopRequiredAlert(suspectPackageName); } }
Fixed — F-02 Critical
3-Strategy process attribution replaces Process.myUid()

The new resolveAttributionUid(filePath) method in UnifiedDetectionEngine tries three strategies in order: (1) parse /data/data/<pkg>/ or /data/user/N/<pkg>/ — deterministic via regex; (2) parse /sdcard/Android/data/<pkg>/ or /storage/emulated/N/Android/data/<pkg>/; (3) foreground non-SHIELD process heuristic. Returns -1 when unattributed. BehaviorCorrelationEngine now accepts the attributed fileUid as a parameter, and when uid==-1 relaxes the network filter to include all recent events rather than returning 0 matches.

Fixed — F-04 Significant (M-02)
SPRT reset boundary prevents pre-reset event contamination

SPRTDetector now tracks lastResetTimestamp (updated on every reset() call) and exposes it via getLastResetTimestamp(). BehaviorCorrelationEngine.correlateFileEvent() now accepts sprtLastResetTimestamp and clamps the file query start with Math.max(windowStart, sprtLastResetTimestamp). Events from a previous detection session can no longer inflate the behavior score of the current window.

Fixed — F-05 + F-06 Significant
Honeyfile names now unpredictable; grace periods now per-file

L-03: Names derived from SHA-256(Build.FINGERPRINT + dir + index + firstInstallTime) — unique per device-app combination, stored in SharedPreferences for restart survival. Ransomware authors knowing SHIELD's naming convention can no longer pre-allowlist the decoys.

L-02: fileCreationTime is now stored per-observer instance from file.lastModified() at construction time. Each honeyfile has its own 5-second grace window, independent of when other honeyfiles were created.

Fixed — F-07 + F-08 Significant
Heartbeat added for watchdog; VPN permission checked before start

L-01: ShieldProtectionService now writes last_heartbeat to SharedPreferences every 30 seconds. On onDestroy(), intentionally_stopped=true is written with synchronous commit() so the watchdog reads it before it can fire. The watchdog can now distinguish a frozen service from an intentionally stopped one.

H-04: startNetworkMonitoring() now calls VpnService.prepare(this) first. If the result is non-null (permission not granted), it routes to MainActivity via REQUEST_VPN_PERMISSION action instead of attempting to start the service and crashing.

Fixed — F-09 + F-10 Significant
Stale attack windows closed on startup; large-file hash optimised

Stale window: checkAndCloseStaleAttackWindow() now runs in the SnapshotManager constructor. If a previous session left an open attack window, it is closed and a stale=true broadcast is sent. Previously a crash mid-attack would cause trackFileChange() to skip all new snapshots forever.

M-03 large-file hash: calculateHash() now uses partial hashing for files >500 MB — first 1 MB + last 1 MB + 8-byte file size fed into SHA-256. Full streaming hash preserved for files ≤500 MB. Prevents snapshot operations from blocking on multi-GB video files.

Bonus Additions Not in v1

SPRT Calibration Telemetry (H-03)

SPRTDetector now logs SPRT_METRIC samples=N llr=X.XXXX state=Y to logcat every 5 samples. This is the instrumentation required to measure the real file modification rate distribution on Android devices before calibrating λ₁=5.0. Without this data, the threshold remained arbitrary.

RecursiveFileSystemCollector — ShieldStats Integration

A new RecursiveFileSystemCollector(rootPath, storage, context) constructor creates a ShieldStats instance shared across all child FileSystemCollector instances in the directory tree, enabling consistent stats tracking across the full recursive monitoring depth.

Fulfilment Check

Is It Fulfilling
Its Intended Behaviour?

Stage-by-stage honest verdict. Updated for v2 — the most significant change is RESTORE moving from a race condition to a confirmed-death-then-restore model.

COLLECT — ✅ Fully Functional (unchanged from v1)

All four data sources correctly wired. FileObserver multi-flag bitmask handling correct. 500ms LRU debounce prevents duplicate flooding. RecursiveFileSystemCollector depth-8 covers subdirectories. MediaStoreCollector re-enabled (H-02) for FUSE-based ransomware. HoneyfileCollector now uses unpredictable names and per-file grace periods.

DETECT — ✅ Working, Attribution Improved, SPRT Still Uncalibrated

The 3-strategy attribution resolver correctly identifies the writing process for files in app-private storage and shared-storage Android/data paths. The foreground-process heuristic covers everything else. The SPRT M-02 timestamp clamp prevents cross-session score contamination. The one remaining gap: λ₁=5.0 files/sec has not been validated against real Android baseline data. SPRT_METRIC logs are now available to perform that measurement.

ISOLATE — Network ✅ Full Block / Process Kill ⚠️ Best-Effort

Emergency VPN blockAllTraffic=true genuinely drops every packet. Process kill remains best-effort — killBackgroundProcesses() is ineffective against foreground services. App Info shortcut and Mode A broadcast are the fallbacks. This is a fundamental Android platform constraint, not a code flaw. No change from v1 — this limitation cannot be resolved without root or Device Policy Manager.

RESTORE — ✅ Race Condition Fixed

The most critical v1 bug is gone. Restore now polls for process death (200ms intervals, 10-second timeout) before initiating file recovery. If the process dies within 10 seconds, restore runs on clean state. If it doesn't, the user is informed to tap Force Stop manually — restore does not race active encryption. The SPRT latch is also cleared on successful death, so post-recovery file activity doesn't immediately re-trigger detection.

Remaining Integration Gaps

Significant — Still Open
Network events stored under SHIELD's UID — behaviour correlation still limited

The file event attribution is now correct (3-strategy resolver). But network events in NetworkGuardService are still stored with android.os.Process.myUid() — SHIELD's UID. The BehaviorCorrelationEngine uid==-1 relaxed-filter path partially mitigates this: when the file UID cannot be resolved, all network events in the 5-second window are returned regardless of UID. But when the file UID is correctly resolved (Strategy 1 or 2), the per-UID network filter still returns 0 matches because no network event carries the ransomware's UID. Pattern 1 (fileCount>5 + networkCount>0) fires correctly only for uid==-1 unattributed events. This is a fundamental VPN platform constraint requiring Mode A for resolution.

// NetworkGuardService — still stores events under SHIELD's own UID: NetworkEvent netEvent = new NetworkEvent( destIp, destPort, protoName, packet.remaining(), 0, android.os.Process.myUid() ← SHIELD's UID, not ransomware's ); // BehaviorCorrelationEngine — relaxed filter for uid==-1 only: if (uid == -1) return all; ← unattributed: returns everything // For uid != -1 (correctly attributed events): returns 0 network matches
Minor — Still Open
android:allowBackup="true" not yet patched in manifest

ConfigAuditChecker correctly flags this as WARN. On a device with USB debugging enabled, adb backup com.dearmoon.shield can extract SharedPreferences including the HMAC integrity baseline and any Keystore-independent data. Fix: set android:allowBackup="false" or configure a data_extraction_rules.xml that excludes sensitive preferences. The TEE-backed HMAC key itself cannot be extracted, but the stored hash values could be used to craft a matching tampered APK.

What the v2 Fixes Actually Change at Runtime

  • Restore no longer races active encryption — files encrypted post-detection are no longer written back onto an actively running ransomware process.
  • SPRT resets correctly after an attack session — post-recovery file activity no longer immediately re-triggers the HIGH RISK alert.
  • Behavior correlation now profiles the correct process — for files in app-private and shared Android/data storage, the UID is deterministic, not SHIELD's own.
  • Honeyfile names are unpredictable per-device — off-device analysis of SHIELD by a malware author cannot pre-determine which filenames to skip.
  • Multiple attacks in sequence are correctly isolated — the SPRT reset boundary prevents previous attack's file events from contributing to a new detection window score.
  • A frozen (non-crashed) service is now detected by the watchdog — heartbeat timeout identifies deadlocked services, not only process terminations.
Remaining Open Issues

Issues Still in the Code
After v2

Three items remain. Two are Mode B issues in the main APK. One is a Mode A pre-merge blocker (not in APK yet).

Significant — Main APK
Issue 1: EXPECTED_SIGNATURE_HASH = null — set before shipping

The constant in SecurityUtils.java is still null. The code now handles this correctly: debug builds log the actual hash and return true; release builds fail closed (return false → checkSecurity() returns false). The service logs a warning and continues running. The TEE-backed ShieldIntegrityManager is the primary protection layer and is independent. The signature check is a secondary layer that is incomplete but not a critical failure in the current dual-layer configuration. Must be filled before a production release.

// SecurityUtils.java — still null, still needs to be set: private static final String EXPECTED_SIGNATURE_HASH = null; // Get the hash: run app once in dev, read logcat for: // "Cert SHA-256: <hash>" → paste that value above
Minor — Main APK
Issue 2: android:allowBackup="true" in AndroidManifest.xml

On a non-rooted device with USB debugging enabled (also flagged by M8 audit), adb backup com.dearmoon.shield can extract private storage. This includes SharedPreferences containing the HMAC integrity baseline in shield_integrity.xml. The Keystore key cannot be extracted, but the stored hash values are accessible. Fix: android:allowBackup="false" in the manifest, or a data_extraction_rules.xml that blocks shared_prefs/shield_integrity.xml specifically.

Critical — Mode A Only (pre-merge blocker)
Issue 3: Unauthenticated kill socket in Mode A daemon

Mode A is not in the shipped APK — this is a blocker for the Mode A merge, not a live vulnerability. The Unix socket at /data/local/tmp/shield_modea.sock uses chmod 0600 and the SELinux policy was explicitly patched to allow untrusted_app to write to shell_data_file sockets. Any installed app knowing the socket path can connect and send a root SIGKILL to an arbitrary PID. No SO_PEERCRED check validates the connecting UID. Fix: read SO_PEERCRED on accept() and verify UID matches the expected app UID before accepting any command.

// modea_daemon.cpp — missing authentication: int client = accept(server_fd, nullptr, nullptr); // ← any process can connect here, no peer check process_kill_command(client, msg_len); // Required fix: struct ucred cred; socklen_t len = sizeof(cred); getsockopt(client, SOL_SOCKET, SO_PEERCRED, &cred, &len); if (cred.uid != EXPECTED_APP_UID) { close(client); continue; }

Previously Critical — Now Resolved

  • waitForProcessDeath(null) — restore raced active encryption. Fixed: package name now threaded through correctly.
  • Process.myUid() as attacker UID — correlation profiled SHIELD itself. Fixed: 3-strategy path-based attribution resolver.
  • SPRT ACCEPT_H1 latch persisted forever — post-recovery re-trigger. Fixed: sprtDetector.reset() called on confirmed process death.
  • Pre-reset event contamination — previous session events inflated current score. Fixed: M-02 SPRT timestamp boundary clamp.
  • Honeyfile names predictable — could be pre-allowlisted. Fixed: L-03 device-fingerprint-derived names.
  • Stale attack window on crash — trackFileChange() skipped all snapshots forever. Fixed: constructor cleanup.
Final Verdict

Final Verdict — v2

The Updated Bottom Line

The v2 update resolves every runtime correctness issue found in the first audit. The three remaining items are: one incomplete constant that needs a value set (signature hash), one manifest flag that needs changing (allowBackup), and one pre-merge blocker in the Mode A subproject that is not in the shipped APK. None of these affect the core detect-isolate-restore loop in a deployed Mode B build.

The most impactful fix — the null package name in waitForProcessDeath — was a regression that would have caused restore to silently race active encryption in every real detection scenario. It is now correct: restore is gated on confirmed process death with a 10-second timeout, and if that times out the user is alerted to Force Stop manually before recovery proceeds.

The process attribution rewrite is architecturally significant. The correlation engine now profiles the actual attacking process for files in app-private and shared-storage paths, rather than building a behaviour profile of SHIELD itself. The uid==-1 fallback for genuinely unattributable paths is the correct, honest representation of what FileObserver can and cannot know without root. The remaining network event attribution gap is a platform constraint that Mode A's eBPF layer exists specifically to solve.

SHIELD is now in a state where the core four-stage loop is both architecturally correct and free of known runtime regressions. The calibration work (λ₁ validation, SPRT_METRIC baseline collection) and the pre-shipping checklist (signature hash, allowBackup, Mode A SO_PEERCRED) are the remaining actionable items before an evaluation-grade deployment.
